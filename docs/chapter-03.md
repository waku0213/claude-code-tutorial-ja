# 第3章 — 実務でよく使うワークフロー

## この章でできるようになること

- バグ修正・機能追加・リファクタリング・テスト生成・コードレビューを Claude Code で完結できる
- 各ワークフローで「どう伝えると精度が上がるか」のコツがわかる
- SIer の現場でそのまま使えるパターンを身につける

---

## 3-1. バグ修正

### 基本パターン：エラーを貼り付けるだけ

バグに遭遇したらエラーメッセージをそのまま貼りましょう。
原因を自分で推測する前に Claude Code に渡すのが最速です。

```
> 以下のエラーが出ている。修正して。

TypeError: Cannot read properties of undefined (reading 'id')
    at app.delete (/todo-api/src/index.ts:34:38)
    at Layer.handle [as handle_request] (express/lib/router/layer.js:95:5)
```

Claude Code はスタックトレースからファイルと行番号を読み取り、該当箇所を確認してから修正案を提示します。

### より精度を上げる：再現手順も添える

```
> 以下のエラーが出ている。修正して。

【エラー】
TypeError: Cannot read properties of undefined (reading 'id')
    at app.delete (/todo-api/src/index.ts:34:38)

【再現手順】
curl -X DELETE http://localhost:3000/todos/999
（存在しない ID を指定したとき）

【期待する動作】
存在しない ID の場合は 404 を返してほしい
```

「期待する動作」を書くと、修正方針まで自動で合わせてくれます。

### 実際にやってみる

`src/index.ts` に意図的にバグを仕込んでみましょう：

```typescript
// DELETE エンドポイント（バグあり版）
app.delete("/todos/:id", (req, res) => {
  const id = parseInt(req.params.id);
  const index = todos.findIndex((t) => t.id === id);
  // バグ: 見つからない場合のチェックがない
  todos.splice(index, 1);
  res.status(204).send();
});
```

このコードで存在しない ID を DELETE すると `splice(-1, 1)` で最後の要素が消えるバグが発生します。

```
> src/index.ts の DELETE エンドポイントで、存在しない ID を指定すると
> 最後の TODO が消えてしまうバグがある。
> 存在しない ID のときは 404 を返すよう修正して。
```

## 3-2. 新機能追加

### 基本パターン：仕様を自然言語で伝える

```
> GET /todos に以下の機能を追加して。
>
> - ?done=true で完了済みのみ取得
> - ?done=false で未完了のみ取得
> - パラメータなしは従来通り全件取得
```

### 複数のファイルにまたがる変更を頼む

機能が大きいときは「何をどのファイルに実装するか」も指示します。

```
> TODO に「優先度」フィールド（high / medium / low）を追加したい。
> 以下の変更をして：
>
> 1. src/types/todo.ts に型定義を追加（なければ作成）
> 2. src/index.ts の POST /todos で priority を受け取る
> 3. priority が指定されない場合は "medium" をデフォルトにする
> 4. GET /todos に ?priority=high のようなフィルタを追加
```

### 「まず設計を相談してから実装」パターン

大きな機能は実装前に相談するのがおすすめです。

```
> TODO にコメント機能を追加したい。
> 設計案（エンドポイント設計・データ構造）を提示して。
> 実装はまだしなくていい。
```

```
（Claude の回答を読む）

> 提案された設計でいいので実装して。
> ただし POST /todos/:id/comments のエンドポイント名を
> POST /todos/:id/notes に変えて。
```

「設計 → 確認 → 実装」の順で進めると、大きくやり直すリスクが下がります。

## 3-3. リファクタリング

### コードの問題点を洗い出してから変更する

```
> src/index.ts を読んで、リファクタリングすべき箇所を教えて。
> 変更はまだしなくていい。
```

回答例：
```
以下の点が改善できます：

1. ルーティングがすべて index.ts に集中している
   → src/routes/todos.ts に切り出すべき

2. 型定義がインラインで書かれている
   → src/types/todo.ts に切り出すべき

3. エラーハンドリングが各ルートに重複している
   → エラーハンドリングミドルウェアに集約すべき
```

```
> 1 と 2 だけ対応して。3 は今回は触らないで。
```

「全部やって」より「これだけやって」の方が安全で確認しやすいです。

### ファイル分割のリファクタリング例

```
> src/index.ts のルーティング部分を src/routes/todos.ts に切り出して。
> 以下の制約を守って：
>
> - src/index.ts では app.use("/todos", todosRouter) の形で呼び出す
> - 既存のエンドポイントの動作は変えない
> - src/index.ts は Express の初期化と listen だけ残す
```

### 型安全性の改善

```
> src/ 以下のファイルで any 型を使っている箇所を探して、
> 適切な型に置き換えて。
> 型が決まらない場合は unknown を使って。
```

## 3-4. テスト生成

### テストケースの洗い出し → 実装の順で進める

```
> src/routes/todos.ts の POST /todos をテストしたい。
> テストすべきケースを列挙して。実装はまだしなくていい。
```

```
（回答例）
1. 正常系: title を渡すと 201 と作成された TODO が返る
2. 異常系: title が空文字のとき 400 が返る
3. 異常系: title が未定義のとき 400 が返る
4. 異常系: title が文字列以外のとき 400 が返る
5. ID の連番: 複数作成したとき id が 1 から順に増える
```

```
> このケースをすべて網羅するテストを書いて。
> テストフレームワークは Jest、HTTP テストは supertest を使って。
> ファイルは src/__tests__/todos.test.ts に作成して。
```

### テストのセットアップも頼む

```
> Jest と supertest をインストールして、package.json の test スクリプトも設定して。
```

### テストが失敗したらそのまま貼る

```
> テストを実行したら以下が失敗した。修正して。

FAIL src/__tests__/todos.test.ts
  ● POST /todos › title が空文字のとき 400 が返る

    expect(received).toBe(expected)
    Expected: 400
    Received: 201
```

## 3-5. コードレビュー

### 差分をレビューしてもらう

コミット前にレビューを依頼するパターンです。

```
> git diff で変更内容を確認して、コードレビューをして。
> 特に以下の観点で見てほしい：
> - バグになりそうな箇所
> - セキュリティ上の問題
> - TypeScript の型定義の甘さ
```

### プルリクエスト前のチェックリスト的に使う

```
> 今の変更をプルリクエストとして出す前に確認したい。
> 以下を確認して：
>
> 1. エラーハンドリングに漏れはないか
> 2. 追加したエンドポイントにバリデーションはあるか
> 3. 既存の処理を壊していないか
```

### SIer 特有：仕様書との整合性チェック

```
> 以下の仕様を満たしているかコードを確認して。
>
> 【仕様】
> - TODO の title は 1〜100 文字
> - done フラグは初期値 false
> - 削除は論理削除（deleted_at フィールドをセット）
>
> コードを読んで、仕様との差異があれば教えて。
```

## 3-6. ワークフローを組み合わせる

実務では複数のパターンを組み合わせて使います。典型的な流れを示します。

### 典型例：新機能を安全に追加する

```
# 1. まず現状を把握させる
> src/ 以下を読んで、現在の todo の型定義と CRUD エンドポイントを教えて

# 2. 設計を相談する
> TODO に期限（deadline）フィールドを追加したい。
> 影響範囲と設計案を教えて。実装はまだしなくていい。

# 3. 設計を確認し、一部を調整して実装する
> 設計の方針でいいので実装して。
> deadline の形式は ISO 8601 文字列（例: "2024-12-31"）で統一して。

# 4. テストを書いてもらう
> 追加した deadline 関連の処理のテストを書いて。

# 5. レビューして完了
> 今回の変更全体をレビューして。問題なければ完了。
```

## よくあるつまずきポイント

### 「修正して」だけだと意図がズレる

原因・期待動作・制約をセットで伝えましょう。

```
# NG: 意図が伝わりにくい
> バリデーションを直して

# OK: 何を・どうしたいかが明確
> POST /todos の title バリデーションが「空文字を通してしまう」バグを修正して。
> 空文字・空白のみの場合は 400 を返してほしい。
```

### テストが実装と一緒に変更されてしまう

```
> テストは変更しないで。src/routes/todos.ts だけ修正して。
```

スコープを明示する一文が重要です（第2章参照）。

### リファクタリング中に動作が変わってしまう

リファクタリング前後でテストを実行して差異がないか確認しましょう。

```
> リファクタリング後に npm test を実行して、すべて通ることを確認して。
> 失敗したら修正もして。
```

### 大きすぎる変更を一度に頼んだ

一度に多くのことを頼むと、変更が大きくなりすぎてレビューが難しくなります。
「1つのプルリクエストに収まるサイズ」を目安に分割しましょう。

```
# 大きすぎる（1回で頼まない）
> TODO アプリにユーザー認証・コメント機能・タグ機能を追加して

# 適切なサイズに分割
> まず TODO の CRUD に型定義を整備するリファクタリングだけやって
（完了 → コミット）
> 次にコメント機能を追加して
（完了 → コミット）
```

## 演習問題

### 演習 3-1（必須）：バグ修正

以下のバグを意図的に仕込み、Claude Code に修正してもらいましょう。

```typescript
// PATCH /todos/:id（バグあり）
app.patch("/todos/:id", (req, res) => {
  const id = parseInt(req.params.id);
  const todo = todos.find((t) => t.id === id);
  // バグ1: 見つからない場合のチェックなし
  // バグ2: req.body の型チェックなし
  todo.title = req.body.title;
  todo.done = req.body.done;
  res.json(todo);
});
```

Claude Code に以下を伝えて修正してもらいます：

```
> PATCH /todos/:id に2つのバグがある。
> 1. 存在しない ID のとき 404 を返していない
> 2. title や done が undefined でも上書きしてしまう
> 修正して。
```

修正後に `curl` で動作確認してください。

### 演習 3-2（必須）：テスト生成

`GET /todos` に対するテストを書いてもらいましょう。
以下のケースを含むよう指示してください：

- TODO が0件のとき空配列が返る
- TODO が複数件あるとき全件返る
- `?done=true` で完了済みのみ返る（先にこのフィルタ機能を追加する必要あり）

### 演習 3-3（発展）：設計相談から実装まで

「TODO にラベル（複数付けられる）機能を追加したい」という要件を持ち込んでみましょう。
実装前に設計を相談し、合意してから実装するフローを体験してください。

完了後、`git log --oneline` で作業履歴を振り返り、どう進めたかを確認してみましょう。

## まとめ

| ワークフロー | 伝えるべきこと | コツ |
|---|---|---|
| バグ修正 | エラー文・再現手順・期待動作 | 推測より先にエラーを貼る |
| 新機能追加 | 仕様・ファイルの場所・制約 | 大きい機能は設計を先に相談 |
| リファクタリング | 問題点の洗い出し → スコープを絞って変更 | 全部一度にやらない |
| テスト生成 | ケースの洗い出し → 実装 | フレームワークを明示する |
| コードレビュー | 見てほしい観点を明示 | プルリク前の習慣にする |
