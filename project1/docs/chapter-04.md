# 第4章 — 効果を高めるプロンプトの書き方

## この章でできるようになること

- 「なぜ意図通りにならないのか」の原因を自分で診断できる
- コンテキストの渡し方を使い分けて、精度を安定させられる
- 大きなタスクを安全に進める分割戦略がわかる
- 失敗したときに素早く立て直せる

---

## 4-1. 「意図通りにならない」原因は9割が情報不足

Claude Code が期待外れの結果を出したとき、まず疑うべきは **指示の中の情報不足** です。
Claude Code はあなたのプロジェクトにいますが、**あなたの頭の中にある前提** は知りません。

### 情報不足の3パターン

**パターン1：目的が伝わっていない**

```
# NG: 何のための変更か不明
> エラーハンドリングを追加して

# OK: 目的がある
> 本番環境でエラーが出たときスタックトレースをクライアントに返さないよう、
> エラーハンドリングを追加して
```

**パターン2：制約が伝わっていない**

```
# NG: 制約がない
> バリデーションを追加して

# OK: 制約が明示されている
> POST /todos のバリデーションを追加して。
> ただし既存のテストは変更しないで。
> また、バリデーションライブラリは使わずに手書きで実装して。
```

**パターン3：完了の定義が伝わっていない**

```
# NG: 完了がどこかわからない
> ルーティングをリファクタリングして

# OK: 完了の定義がある
> src/index.ts のルーティングを src/routes/todos.ts に切り出して。
> 完了の定義：
> - npm test がすべて通る
> - src/index.ts に残るのは app.use() と app.listen() だけ
```

---

## 4-2. コンテキストを渡す3つの方法

「コンテキスト」とは Claude Code が正確な回答を出すために必要な背景情報のことです。
渡し方には3つのレベルがあります。

### レベル1：CLAUDE.md（永続的なコンテキスト）

起動のたびに自動で読み込まれます。**毎回伝え直していることはここに書く。**

```markdown
# CLAUDE.md に書くべき内容の例

## テストについて
- Jest + supertest を使用
- テストファイルは src/__tests__/ 以下に配置
- テスト実行: npm test

## コーディング規約
- any 型禁止
- エラーレスポンスは { error: string } の形式に統一
- 環境変数は src/config.ts 経由で読む（process.env を直接使わない）

## 触ってはいけないファイル
- .env は直接編集しない
- package-lock.json は手動で編集しない
```

> **チェック：** 「Claude に3回以上同じことを言った」と気づいたら CLAUDE.md に書く。

### レベル2：会話の冒頭（セッションのコンテキスト）

そのセッションでのみ有効な情報を最初に渡します。

```
> 今日のタスクを始める前に状況を共有する。
>
> 現在 feature/add-labels ブランチで作業中。
> src/routes/todos.ts にラベル機能を追加している途中で、
> POST /todos/:id/labels と DELETE /todos/:id/labels は実装済み。
> 今から GET /todos/:id/labels を追加したい。
```

### レベル3：指示の中（その場のコンテキスト）

1回限りの指示に必要な情報はその場で渡します。

```
> src/routes/todos.ts を読んで。
> このファイルは先週のリファクタリングで index.ts から切り出したもの。
> GET /todos/:id/labels を追加してほしいが、
> 既存の GET /todos/:id の処理パターンに合わせて実装して。
```

## 4-3. 「何をしてほしいか」より「何を達成したいか」

Claude Code に頼むとき、**操作の指示** より **達成したい状態** を伝える方が良い結果になります。

### 操作指示 vs 達成状態

| 操作の指示（避ける） | 達成状態（推奨） |
|---|---|
| `if 文を追加して` | `存在しない ID のとき 404 を返すようにして` |
| `try-catch を書いて` | `データベースエラーをクライアントに漏らさないようにして` |
| `型を修正して` | `コンパイルエラーをすべて解消して` |
| `テストを書いて` | `追加したバリデーションが正しく動くことをテストで保証して` |

### 理由

操作を指示すると、Claude Code は「その操作をすること」が目的になります。
達成状態を伝えると、最適な実装方法を Claude Code が選んでくれます。

### 例：バリデーションの追加

```
# 操作の指示（結果が狭くなる）
> title の長さを確認する if 文を POST /todos に追加して

# 達成状態（最適な実装を選んでくれる）
> POST /todos の title について以下を保証して：
> - 必須（undefined・null・空文字はエラー）
> - 1〜100文字の範囲
> - バリデーション失敗時は 400 と具体的なエラーメッセージを返す
```

## 4-4. タスクの分割戦略

大きなタスクを一度に頼むほど、途中で方向がズレたときのやり直しコストが増えます。
「**確認できるサイズ**」に分割するのが鉄則です。

### 分割の基準：1コミットに収まるか

1つの指示で変更するコードが「1コミットとして意味のある単位」かどうかが目安です。

```
# 大きすぎる（一度に頼まない）
> TODO アプリをデータベース（SQLite）対応にして

# 適切に分割した例

# Step 1
> sqlite3 と better-sqlite3 をインストールして、
> src/db.ts にデータベース接続の初期化コードを書いて

# Step 2（Step 1 を確認・コミット後）
> TODO のインメモリ配列を SQLite に置き換えて。
> src/routes/todos.ts の GET /todos だけ先に対応して

# Step 3（Step 2 を確認・コミット後）
> 残りの POST, PATCH, DELETE も SQLite に対応させて

# Step 4（Step 3 を確認・コミット後）
> データベース操作のテストを書いて
```

### 「一気にやって」が有効なケース

逆に、一気にやってもらった方が良いケースもあります：

- **スコープが明確で小さい**（「このファイルの any を全部 unknown にして」）
- **自動で確認できる**（「テストを書いて、npm test が通ることを確認して」）
- **失敗してもすぐ戻せる**（git 管理下で、変更が1ファイル以内）

## 4-5. 失敗したときの立て直し方

### パターン1：出てきたコードが意図と違う

まず **何が違うか** を具体的に伝えて修正を頼みます。

```
> 出てきたコードで問題が2点ある。
>
> 1. title のバリデーションで空文字を通してしまっている。
>    `title.length === 0` のチェックを追加して。
>
> 2. エラーレスポンスの形式が { message: ... } になっているが、
>    このプロジェクトの規約は { error: ... } に統一している。
>    修正して。
```

「違う」だけを伝えるより、**どこが・何が・なぜ違うか** を伝えると一発で直ります。

### パターン2：変更が大きくなりすぎた

承認前なら `n` で全部断って指示を出し直します。

```
> 承認前に確認したが変更が大きすぎる。
> 一度やり直す。
> 今回は src/routes/todos.ts だけ変更して。
> src/index.ts と src/types/ には触らないで。
```

承認後なら git で戻します：

```bash
git checkout .
```

### パターン3：途中まで進んで止まった・迷走した

`/clear` でリセットして、改めて **現在の状態から** 指示を出します。

```
> /clear

> 現在の状態を確認したい。src/ 以下を読んで。
```

```
> ありがとう。状況を整理する。
> src/routes/todos.ts は変更済みで、GET・POST・DELETE は実装済み。
> PATCH だけ未実装の状態。
> PATCH /todos/:id を追加して。仕様は以下の通り...
```

「今どこにいるか」を Claude Code に把握させてから指示するのがポイントです。

### パターン4：何度やっても同じ間違いを繰り返す

CLAUDE.md に「やってはいけないこと」を書きます。

```markdown
# CLAUDE.md に追記

## 注意事項（過去のミスから）
- エラーレスポンスは必ず { error: string } の形式にする（{ message: ... } は使わない）
- テストファイルは実装ファイルと一緒に変更しない
- req.body の値は必ず型チェックしてから使う
```

次回以降は同じ指摘が不要になります。

## 4-6. 高度なテクニック

### ロールを指定する

どういう立場で回答してほしいかを伝えると、出力のスタイルが変わります。

```
# コードレビュアーとして
> シニアエンジニアの立場でこのコードをレビューして。
> 特にセキュリティとパフォーマンスの観点で指摘して。

# 新人へのレビューコメントとして
> このコードの改善点を、新卒エンジニアに説明するように教えて。
> 理由も添えて。
```

### 出力形式を指定する

```
# 箇条書きで
> このAPIの問題点を箇条書きで5つ以内にまとめて

# 表形式で
> エンドポイント一覧をMarkdownの表形式でまとめて

# コードのみ
> 修正後のコードだけ出して。説明は不要。
```

### 段階的な確認（Think Step by Step）

複雑な問題は手順を踏んで考えてもらいます。

```
> src/routes/todos.ts を SQLite に移行する前に、
> 影響を受けるファイルと変更内容を順番に整理して。
> コードは書かなくていい。段階的に考えて。
```

## よくあるつまずきポイント

### 毎回長い前置きを書くのが面倒

**→ CLAUDE.md に移動する。** 前置きが必要だと感じた瞬間が「CLAUDE.md に書く」サインです。

### 指示が長すぎてまとまらない

複雑に感じるなら、タスク自体が大きすぎる可能性があります。分割して1つずつ頼みましょう。

### うまく伝わらなかったが何が悪かったかわからない

Claude Code に直接聞くのが最速です。

```
> さっきの指示で意図通りの結果にならなかった。
> 指示のどこが曖昧だったか教えて。
```

## 演習問題

### 演習 4-1（必須）：CLAUDE.md を育てる

これまでの章でやり取りをする中で、「毎回伝えたこと」「毎回説明したこと」を思い出し、
CLAUDE.md に3項目以上追記しましょう。

追記後に `/clear` して再起動し、「このプロジェクトのエラーレスポンスの形式は？」のような質問をして、
CLAUDE.md から正確に答えられるか確認してください。

### 演習 4-2（必須）：達成状態で伝える練習

以下の「操作の指示」を「達成状態の指示」に書き直してみましょう。
書き直した後、実際に Claude Code に投げて結果の差を比較してください。

1. 「`try-catch` を追加して」
2. 「ログを出力するようにして」
3. 「型を直して」

### 演習 4-3（発展）：タスクの分割計画

「TODO に締め切り通知機能を追加する（締め切り前日にコンソールに表示）」という要件を
**3〜5ステップに分割**し、各ステップの指示文を書いてみましょう。
書き終えたら実際に順番に実行してみてください。

## まとめ

| 学んだこと | ポイント |
|---|---|
| 意図通りにならない原因 | 目的・制約・完了の定義が抜けている |
| コンテキストの渡し方 | CLAUDE.md（永続）/ 会話冒頭（セッション）/ 指示の中（その場） |
| 伝え方の原則 | 操作より達成状態を伝える |
| タスクの分割 | 1コミットに収まるサイズが目安 |
| 失敗の立て直し | 何が違うかを具体的に → git で戻す → `/clear` してリセット |
