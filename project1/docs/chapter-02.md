# 第2章 — 基本的な使い方

## この章でできるようになること

- ファイルやディレクトリを Claude Code に正確に渡す方法がわかる
- 変更の承認・拒否・取り消しを使い分けられる
- 会話の流れをコントロールして、意図通りの結果を引き出せる

---

## 2-1. Claude Code の画面構成

`claude` を起動すると以下のプロンプトが表示されます。

```
claude-code v1.x.x  |  todo-api
> 
```

操作中に知っておくべき表示：

| 表示 | 意味 |
|---|---|
| `>` | 入力待ち |
| `◐ thinking...` | Claude が考えている |
| `✎ src/index.ts` | ファイルを読み込んでいる |
| `? Apply this change? (y/n)` | 変更の承認を求めている |
| `✓ Done` | 作業完了 |

### 終了方法

```
> /exit
```

または `Ctrl + C` で終了できます。

---

## 2-2. コードを「読んでもらう」技術

### 基本：ファイル名を指定して聞く

```
> src/index.ts を読んで、このファイルの処理の流れを説明して
```

Claude Code はファイルを自動で読み込み、内容をふまえて回答します。
**コードをコピペする必要はありません。**

### ディレクトリごと渡す

```
> src/ ディレクトリ全体を読んで、全体的なアーキテクチャを教えて
```

ファイル数が多い場合は自動的に重要なファイルを優先して読みます。

### 複数ファイルを同時に渡す

```
> src/index.ts と src/routes/todos.ts を比べて、重複している処理があれば教えて
```

### 「読んで」を先に頼む

何かを変更してもらう前に、まず読んでもらう習慣をつけると精度が上がります。

```
# 悪い例：いきなり変更を頼む
> src/index.ts にバリデーションを追加して

# 良い例：読んでから変更を頼む
> src/index.ts を読んで、POST /todos のバリデーションが足りない箇所を教えて
→ 回答を確認する
> じゃあそこにバリデーションを追加して
```

---

## 2-3. 変更の承認・拒否

Claude Code がファイルを書き換えようとするとき、デフォルトでは必ず確認を求めます。

```
? Write to src/index.ts? (y/n/d/s)
```

| キー | 動作 |
|---|---|
| `y` | 変更を承認して適用する |
| `n` | この変更を拒否する（別の方法を試してもらえる） |
| `d` | 変更内容の diff を表示してから判断する |
| `s` | この変更をスキップして次の変更へ進む |

### diff を確認してから承認する習慣

**`d` を押して diff を確認してから `y` を押す** のが安全な使い方です。

```diff
? Write to src/index.ts? (d)

- app.post("/todos", (req, res) => {
-   const { title } = req.body;
-   const todo = { id: nextId++, title, done: false };
+ app.post("/todos", (req, res) => {
+   const { title } = req.body;
+   if (!title || typeof title !== "string") {
+     return res.status(400).json({ error: "title is required" });
+   }
+   const todo = { id: nextId++, title: title.trim(), done: false };

Apply this change? (y/n)
```

意図通りかどうかをここで確認し、問題なければ `y` を押します。

## 2-4. 変更の取り消し

### パターン1：承認前に気づいた場合

確認プロンプトで `n` を押すだけです。ファイルは変更されません。

```
? Write to src/index.ts? → n
```

### パターン2：承認後に気づいた場合

**git があれば一瞬で戻せます。** これが「作業前に git commit」を勧める理由です。

```bash
# 変更されたファイルを確認
git diff

# 特定ファイルだけ戻す
git checkout src/index.ts

# すべての変更を戻す
git checkout .
```

### パターン3：会話の中で「やっぱり前の状態に戻して」と頼む

```
> さっきの変更は取り消して、元の実装に戻して
```

Claude Code は会話の履歴を持っているので、前の状態に戻すコードを書いてくれます。
ただし git を使った方が確実です。

> **鉄則：** Claude Code を使い始める前に必ず `git add . && git commit -m "wip"` しておく。
> これだけで「どんな変更も取り消せる」安心感が手に入ります。

## 2-5. 会話を続ける・リセットする

### 会話を続ける

同じ `claude` セッションの中では、前のやり取りの文脈が引き継がれます。

```
> src/index.ts を読んで
→ （Claude が読む）

> じゃあ DELETE エンドポイントを追加して
→ （すでに読んでいるので、改めて読み込まなくてよい）

> 型定義が足りない気がする。修正して
→ （前の変更内容も把握した上で対応してくれる）
```

### 会話をリセットする

```
> /clear
```

これで会話履歴がリセットされます。別のタスクに切り替えるときに使います。
ファイルの変更は消えません。あくまで「会話の記憶」をリセットするだけです。

### セッションをまたいで文脈を引き継ぐ

`claude` を再起動しても CLAUDE.md の内容は毎回読み込まれます。
「毎回伝え直していること」は CLAUDE.md に書いておきましょう。

```
> /clear してから再起動したけど、このプロジェクトのテスト実行コマンドは？
→ CLAUDE.md に書いてあれば答えてくれる
```

## 2-6. よく使う指示パターン

実際の作業でよく使う指示の型をまとめます。コピーして使ってください。

### 読解系

```
# ファイルの概要を知りたい
> src/routes/todos.ts を読んで、このファイルの責務を一言で教えて

# 特定の処理を追いたい
> POST /todos のリクエストがどう処理されるか、コードを追って説明して

# 依存関係を知りたい
> src/index.ts はどのモジュールに依存しているか一覧にして
```

### 変更系

```
# 機能追加
> GET /todos に ?done=true でフィルタリングできるクエリパラメータを追加して

# バグ修正（エラーを貼る）
> 以下のエラーが出ている。修正して。
> TypeError: Cannot read properties of undefined (reading 'id')

# リファクタリング
> src/index.ts のルーティングを src/routes/todos.ts に切り出して
```

### 確認系

```
# 変更前にレビューしてもらう
> この実装で問題ないか確認して。特にエラーハンドリングを重点的に見て

# テストを書いてもらう前に確認
> POST /todos のテストを書く前に、テストすべきケースを列挙して
```

## よくあるつまずきポイント

### 意図と違う変更をされた

まず指示が曖昧だった可能性を疑いましょう。

```
# 曖昧な指示
> バリデーションを追加して

# 具体的な指示
> POST /todos の title フィールドが空文字や undefined のとき
> 400 を返すバリデーションを追加して
```

承認前に diff を確認する（`d` キー）習慣もつけましょう。

### 関係ないファイルまで変更された

```
> src/index.ts だけを変更して。他のファイルは触らないで
```

スコープを明示する一文を加えると防げます。

### Claude が「ファイルが見つからない」と言う

カレントディレクトリを確認してください。
`claude` はコマンドを打ったディレクトリを起点にします。

```bash
# プロジェクトルートから起動する
cd todo-api
claude
```

### 長い作業の途中で止まった

`Ctrl + C` で中断して、どこまで進んだかを確認します。

```
> さっきの作業は /src/routes/todos.ts を追加したところで止まった。
> 残りの作業（src/index.ts の修正）だけ続けて
```

## 演習問題

### 演習 2-1（必須）：読解の練習

`src/index.ts` を Claude Code に読んでもらい、以下を回答してもらいましょう：

1. 「このAPIのエンドポイントを一覧にして」
2. 「エラーハンドリングが不十分な箇所はどこか」
3. 「TypeScript の型定義として改善できる点はあるか」

diff を見ながら答えを確認し、納得できる回答かどうかを評価してみてください。

### 演習 2-2（必須）：変更と取り消しの練習

1. `GET /todos/:id`（ID指定で1件取得）を追加してもらう
2. 変更前に `git commit` しておく
3. diff（`d`）を確認してから承認する
4. `git diff HEAD~1` で変更内容を確認する
5. `git checkout .` で変更を取り消し、元に戻ることを確認する
6. 改めて「今度は取り消さず」同じ変更を追加する

### 演習 2-3（発展）：スコープを制御する

以下の指示を試して、「意図通りに動くか」を確認してみましょう：

```
> src/routes/ ディレクトリを作り、todos のルーティングを切り出して。
> src/index.ts は import 部分だけ変更して、それ以外は触らないで。
```

変更が意図通りでなければ `n` で拒否し、指示を言い直してみてください。

## まとめ

| 学んだこと | ポイント |
|---|---|
| ファイルの渡し方 | ファイル名やディレクトリ名を自然な日本語で書けばOK |
| 変更の承認 | `d` で diff 確認 → `y` で承認が基本ルーティン |
| 変更の取り消し | `git checkout .` が最速・最確実 |
| 会話のリセット | `/clear` で履歴リセット、ファイルは消えない |
| 指示の精度を上げる | 曖昧な指示 → 具体的な指示、スコープを明示する |
